## 成本不对称：验证器付费，矿工免费

验证器承担**所有 LLM 推理成本**——他们运行 ClawBench 情景来评估每个矿工的策略包。矿工提交静态策略包（JSON）并且每个纪元支付零推理成本；他们唯一的成本是注册和研发迭代。

这是有意设计的：矿工在*智能*（更好的提示/策略）上竞争，而不是在计算资源上。

## 验证器成本模型

每个评估间隔（24小时），验证器评估标记为需要重新评估的活跃矿工——要么他们的 `pack_hash` 已更改，要么自上次评估以来已超过 `eval_interval`：

```
每个评估的回合数 = 情景数(7) × 每个情景运行1次 = 每个矿工7个回合
每天最大评估次数 = 24小时 / 评估间隔(24小时) = 1次
每天总回合数 = 标记的矿工数 × 7
```

> **EMA 累积**：验证器定期重新评估策略包（即使未更改）以累积每个情景的 EMA 样本并平滑 LLM 非确定性。限制为每个矿工每个评估间隔最多评估1次。

**每个回合的观察成本**（跨5个情景的中位数，多轮代理对话）：

代理通常根据情景复杂性每个回合进行2-8次 LLM 调用。观察到的成本范围：**$0.05–$0.13** 每个回合，中位数 **~$0.08**。

## 每日成本预测

指定模型：通过 OpenAI 兼容 API 使用 `GLM-5`。每个回合成本 ≈ **$0.08**（观察到的中位数）。

**所有验证器必须使用指定模型。** 这是共识要求：如果验证器使用不同的模型，代理会产生不同的工具调用序列，导致不同的评分结果和验证器在分数上的分歧。使用错误的模型会使您的验证器失去共识并面临被 Yuma 降权的风险。

### 最坏情况：所有矿工每个评估间隔都重新评估

| 活跃矿工数 | 每天回合数 | 每日成本 | 每月成本 |
|:-------------:|:------------:|:----------:|:------------:|
| 5 | 35 | **$3** | **$84** |
| 14 | 98 | **$8** | **$235** |
| 30 | 210 | **$17** | **$504** |
| 64 | 448 | **$36** | **$1,075** |
| 128 | 896 | **$72** | **$2,150** |
| 256 | 1,792 | **$143** | **$4,301** |

**最坏情况公式**：`每日成本 ≈ 矿工数 × 1次评估 × 7个回合 × $0.08 = 矿工数 × $0.56/天`。

实际上，只有 `pack_hash` 发生变化的矿工会立即重新评估。稳定的策略包每个评估间隔（24小时）重新评估一次以进行 EMA 累积。

## 矿工成本模型

| 成本项目 | 估算 |
|-----------|----------|
| 策略迭代（提示调优） | 仅工程师时间 |
| 通过 ClawBench 进行本地测试 | ~$0.08/回合 × ~50次测试运行 ≈ **$4/次迭代** |
| GitHub 仓库托管 | 免费 |
| Bittensor 注册 | ~200 TAO（一次性） |
| **持续运营成本** | **~$0/月** |

## 成本降低杠杆

1. **速率限制**（内置）：每个矿工每个评估间隔（24小时）最多评估1次，无论矿工更新承诺的频率如何。防止快速提交耗尽 API 预算
2. **EMA 收敛**：一旦策略包的 EMA 分数稳定，重新评估的价值会递减。未来优化：当 EMA 方差低于阈值时跳过重新评估
3. **提示缓存**：Anthropic 提示缓存节省约80%的输入令牌（相同情景的测试数据在不同运行中是相同的）

## 可持续性

验证器经济性依赖于 alpha 收益（可转换为 TAO）超过 LLM 成本。

验证器赚取**子网 alpha**，而不是直接的 TAO。Alpha 可以通过子网的流动性池以市场确定的汇率兑换为 TAO。当前 SN11 alpha 价格：~$2.64（≈0.015 TAO，按 ~$180/TAO 计算）。

```
估算的 alpha 收益（中等质押 ~5k TAO，~10% 验证器权重）：
  ~295 alpha/天 ≈ 按当前池汇率约4 TAO ≈ $720/天

示例（30个矿工，最坏情况所有矿工每个间隔都重新评估）：
  每日成本：30 × $0.56 = $17/天
  每日收入：~$720/天（alpha，按当前池汇率）
  净利润：~$703/天（~98% 利润率）

示例（30个矿工，典型日）：
  每日成本：~$7-14/天（大多数策略包未更改）
  每日收入：~$720/天
  净利润：~$706-713/天
```

**在当前费率下**，TrajectoryRL 验证器具有高盈利性：

| 情景 | 每日成本（最坏情况） | 每日收入（~$720 alpha） | 每月利润 |
|----------|:----------:|:---------------------------:|:--------------:|
| 30 个矿工 | $17 | $720 | **$21,090** |
| 64 个矿工 | $36 | $720 | **$20,520** |
| 128 个矿工 | $72 | $720 | **$19,440** |
| 256 个矿工 | $143 | $720 | **$17,310** |

即使在256个矿工（最坏情况，所有矿工每个评估间隔都重新评估）的情况下，LLM 成本仅占验证器 alpha 收入的 **~20%**。

**盈亏平衡分析**：在256个矿工（$143/天最坏情况成本）的情况下，验证器在当前 alpha 费率下极具盈利性。在验证器变得无利可图之前，alpha-TAO 池汇率需要下降约80%。注意：这些数字会随着池汇率和子网需求波动。

## 权重设置

每个验证器基于自己的评估数据独立设置权重。没有共享的分数仓库或链下共识机制。

每个 tempo（约72分钟），验证器：
1. 检查资格：每个矿工必须通过所有情景的所有安全性和正确性检查
2. 按成本排名合格的矿工（成本最低的获胜），使用 EMA 平滑的成本
3. 应用先发者保护：挑战者必须比现任者便宜 ≥10% 才能取代
4. 通过当前元图将矿工热键映射到 UID
5. 通过提交-揭示在链上设置权重

跨验证器共识完全由 **YC3（Yuma Consensus 3）** 与 **Liquid Alpha** 在链上处理，后者根据验证器之间的一致程度动态调整每个质押的学习率。

有关资格门、成本排名和 YC3 配置的完整详细信息，请参阅 [INCENTIVE_MECHANISM.md — 验证器共识](INCENTIVE_MECHANISM.md#validator-consensus)。

## 启动选项

### `EVAL_ON_STARTUP`

在验证器启动时立即触发一个完整的评估周期，而不等待下一个预定的 UTC 00:00 窗口。在重启或手动干预后不希望等到午夜时很有用。

```bash
docker compose -f docker/docker-compose.validator.yml --env-file .env.validator \
  up -d -e EVAL_ON_STARTUP=1 validator
```

启动评估完成后，验证器恢复其正常的每日计划（UTC 00:00）。如果启动评估在午夜前完成，每日评估仍将照常在 00:00 触发。

## 自动更新

验证器镜像托管在 GHCR（GitHub 容器注册表）上。Watchtower 与验证器一起运行，并在更新推送到 `prod` 时自动拉取新镜像，通常在5分钟内。

无需手动操作——Watchtower 处理镜像拉取、容器重启和清理。

### 验证自动更新

```bash
# 检查 Watchtower 日志
docker compose -f docker/docker-compose.validator.yml logs watchtower

# 检查当前验证器镜像
docker inspect trajectoryrl_validator --format '{{.Image}}'
```

### 手动更新（如果不使用 Watchtower）

```bash
docker compose -f docker/docker-compose.validator.yml --env-file .env.validator pull validator
docker compose -f docker/docker-compose.validator.yml --env-file .env.validator up -d validator
```
