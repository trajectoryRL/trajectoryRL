## 成本不对称：验证者付费，矿工免费

验证者承担**所有 LLM 推理成本** —— 他们需要针对每个矿工的策略包运行 ClawBench 剧集（episodes）。矿工仅提交静态策略包（JSON格式），并且每个 epoch 的推理成本为零；他们只需承担注册费用和研发迭代的时间成本。

这是有意设计的：矿工在**智能**上进行竞争（更好的提示词/策略），而不是在算力上比拼。

## 验证者成本模型

在每个评估周期（eval_interval，目前为 24 小时）内，验证者会对那些被标记为需要重新评估的活跃矿工进行测试——触发条件是 `pack_hash` 发生变化，或者距上次评估已满一个评估周期：

```
每个矿工的评估剧集 = 7 个场景 × 各运行 1 次 = 7 个剧集
每天最大评估次数 = 24h / eval_interval(24h) = 1 次
每天总评估剧集 = 被标记的矿工数 × 7
```

> **EMA 样本累积**：验证者会定期重新评估策略包（即使没有更改），以累积各场景下的指数移动平均（EMA）样本，从而平滑处理 LLM 的不确定性。频率限制为每个评估周期内最多对同一矿工评估 1 次。

**每个剧集的实测成本**（跨 5 个场景的中位数，涉及多轮 Agent 对话）：

在一个剧集中，Agent 通常会进行 2–8 次 LLM 调用，具体取决于场景的复杂度。实测成本范围为**大约每剧集 $0.05–$0.13**，中位数为**~$0.08**。

## 每日成本预测

指定模型：通过 OpenAI 兼容 API 调用的 `GLM-5`。每个剧集的成本约 **$0.08**（实测中位数）。

**所有验证者必须使用指定模型。** 这是一项共识要求：如果验证者使用不同的模型，Agent 就会生成不同的工具调用序列，导致评分结果大相径庭，进而在分数共识上出现分歧。使用错误的模型将使你的验证者脱离共识，并面临被 Yuma 系统降权的风险。

### 最坏情况：所有矿工均在每个 eval_interval 接受重评

| 活跃矿工数量 | 每天评估剧集 | 每日成本 | 每月成本 |
|:-------------:|:------------:|:----------:|:------------:|
| 5 | 35 | **$3** | **$84** |
| 14 | 98 | **$8** | **$235** |
| 30 | 210 | **$17** | **$504** |
| 64 | 448 | **$36** | **$1,075** |
| 128 | 896 | **$72** | **$2,150** |
| 256 | 1,792 | **$143** | **$4,301** |

**最坏情况公式**：`每日成本 ≈ 矿工数量 × 1 次评估 × 7 个剧集 × $0.08 = 矿工数量 × $0.56 / 天`。

实际上，只有 `pack_hash` 改变的矿工才会立即被重新评估。未改变的稳定策略包则会在每个 eval_interval（24 小时）周期时评估一次，用于 EMA 分数累积。

## 矿工成本模型

| 成本项 | 预估说明 |
|-----------|----------|
| 策略迭代（提示词调优） | 仅限工程师的研发时间 |
| 基于 ClawBench 进行本地测试 | ~$0.08/剧集 × ~50 次测试运行 ≈ **每轮迭代 $4** |
| GitHub 托管仓库 | 免费 |
| Bittensor 注册费 | ~200 TAO（一次性） |
| **持续运维成本** | **每月 ~$0** |

## 成本优化手段

1. **频率限制**（内置）：无论矿工更新策略的频率多高，在一个 eval_interval（24h）内最多评估 1 次。防止频繁提交导致 API 预算耗尽。
2. **EMA 收敛**：一旦策略包的 EMA 得分趋于稳定，重新评估带来的价值就会递减。未来的优化方案：当 EMA 波动低于特定阈值时，跳过重新评估。
3. **提示词缓存**（Prompt caching）：Anthropic 提示词缓存技术能够在大约 80% 的输入 token 上节省成本（针对相同场景的不同运行，其固定数据是一致的）。

## 可持续性与收益

验证者的经济情况取决于 Alpha 代币收益（可兑换为 TAO）是否超过 LLM API 的调用成本。

验证者赚取的是**子网 Alpha**，而不是直接获取 TAO。Alpha 可以通过对应子网的流动性池以市场决定的汇率兑换为 TAO。 当前 SN11 的 Alpha 价格：~$2.64 (按 ~$180/TAO 计算 ≈0.015 TAO)。

```
预估 Alpha 收益（中等质押量 ~5k TAO，大约拥有 10% 的验证者权重）：
  每天赚取 ~295 alpha ≈ 4 TAO 当量 (按当前池子汇率算) ≈ $720/天

示例（假设 30 名矿工，并且为最坏情况：所有人在每个周期内均被重评）：
  每日成本： 30 × $0.56 = $17/天
  每日营收： ~$720/天（Alpha 收益，当前汇率）
  预估净利： ~$703/天（~98% 利润率）

示例（假设 30 名矿工，普通日常）：
  每日成本： ~$7-14/天（大多数策略包未发生改变）
  每日营收： ~$720/天
  预估净利： ~$706-713/天
```

**在当前汇率下**，TrajectoryRL 验证者的盈利能力极高：

| 场景 | 每日成本 (最坏情况) | 每日营收 (~$720 alpha) | 每月净利 |
|----------|:----------:|:---------------------------:|:--------------:|
| 30 矿工 | $17 | $720 | **$21,090** |
| 64 矿工 | $36 | $720 | **$20,520** |
| 128 矿工 | $72 | $720 | **$19,440** |
| 256 矿工 | $143 | $720 | **$17,310** |

即使在有 256 名矿工的极端情况（所有矿工在每个周期都被重新评估），LLM 成本也仅占验证者 Alpha 营收的 **~20%**。

**盈亏平衡点分析**：在有 256 名矿工的最差成本情境（$143/天）中，以目前的 Alpha 价格，验证者也是高度盈利的。 Alpha 兑换 TAO 的汇率下跌约 80%，验证者才会从盈利转为亏损。注意：这些数据会随着流动性池的汇率和该子网的实际需求而波动。

## 权重设置 (Weight Setting)

每个验证者都会基于自身的评估数据独立设定对矿工的评价权重。系统没有共享分数的统一仓库，也没有链下的共识机制。

每个 tempo（大约 72 分钟），验证者会执行以下流程：
1. 检查合格性：矿工必须在所有情境中，不仅通过安全检查而且通过正确性检查。
2. 依据成本对合格的矿工排序（$/剧集最低者获胜），使用基于 EMA 平滑处理的成本数据。
3. 应用先发者保护：挑战者的成本必须比现任保持者低 ≥10%，才能取而代之。
4. 在当前的 metagraph 中将 miner hotkeys 映射到 UID。
5. 通过 commit-reveal 的方式在链上设置权重。

跨验证者的共识计算是完全通过链上的 **YC3 (Yuma Consensus 3)** 和 **Liquid Alpha** 机制来处理的，它会动态地根据验证者的共识情况来调整每一个 bond 当中的学习率。

有关质量门槛检测、基于成本的排名和 YC3 配置等全面详细的阐述，请参阅 [INCENTIVE_MECHANISM.md — 验证者共识机制](INCENTIVE_MECHANISM.md#validator-consensus)。

## 启动选项参数

### `EVAL_ON_STARTUP`

能在验证者启动时立刻执行一次完整的评测循环，而不需要等到预设的下一个 UTC 00:00 评估触发点。此配置很适用于机器刚刚经历了重启或是你完成了一次人为手动干预，进而不想等到午夜才进行评定的场景下。

```bash
docker compose -f docker/docker-compose.validator.yml --env-file .env.validator \
  up -d -e EVAL_ON_STARTUP=1 validator
```

当特派的这场启动评估完成时，验证者程序将同样重新回归其例行运作，后续它仍然定在 UTC 00:00 的时段正常响应。如果在午夜 00:00 之前此番评测已顺畅告终，那常规的午夜那批排程仍然按既定规则触发。

## 自动更新

验证者的镜像文件当前寄托并在 GitHub 容器镜像库（GHCR）得到托管。系统侧采用一款名为 Watchtower 的常驻并跑的监控进程服务，主要它会负责自动抓取在项目往 `prod` 的环境所触发发布的各个版本的更新。这期间的大体时间顺延并没超多大概五分钟左右的空隙。

在这个自动流转的程序里一切全自动操作—你并不被期望须在此作其它任何手工应对干涉，从抓获取镜像文件直到去关闭并剔除那些历史过期无效的镜像等事项的接棒，Watchtower 均都会接手。

### 检验自动更新的情况

```bash
# 查询并检查 Watchtower 日志情况
docker compose -f docker/docker-compose.validator.yml logs watchtower

# 校对验证者在环境内的当选镜像信息
docker inspect trajectoryrl_validator --format '{{.Image}}'
```

### 手动执行更新升级 (如果您未使用 Watchtower)

```bash
docker compose -f docker/docker-compose.validator.yml --env-file .env.validator pull validator
docker compose -f docker/docker-compose.validator.yml --env-file .env.validator up -d validator
```
